# Offline resilience — гарантии «месяц без связи и больше»

**Дата:** 2026-05-18 · **MDM-сервер:** ≥ 0.17.0 · **Outpost-Android:** rc42 b37+

Этот документ фиксирует **серверные гарантии** для устройств, оторванных от сети на длительное время (≥30 дней). Поведение client-side (локальный LLM, knowledge.db, telemetry queue на устройстве) — отдельная зона ответственности AR Hud команды.

## TL;DR

- **Устройство online хотя бы раз в 45 дней** → session живёт **вечно**, никаких re-enroll'ов не требуется.
- **Устройство offline до 90 дней** → возвращается online → session действительна → /sync работает, накопленные admin push-команды доставляются разом.
- **Устройство offline > 90 дней без единого /sync** → session expires → требуется **полный re-enroll** через admin (новый QR-payload).
- **Encrypted distribution blob'ы**: лежат на диске пока их `expires_at` не пройдёт + 7 дней grace. Без `expires_at` — навсегда.

## Что выдерживает offline

### Device session (90-дневный token + sliding refresh)

`DEVICE_TOKEN_TTL_SECS = 60×60×24×90 = 7,776,000 секунд` (90 дней). При создании сессии в `routes/enrollment.rs::enroll` server создаёт row в `sessions` с `expires_at = now() + 90 дней`.

**Sliding refresh (v0.17):** при каждом `POST /api/v1/sync` server вызывает `session::refresh_if_aging_for_subject(kind='device', subject_id=device.id, full_ttl=90 дней, threshold=50%)`. Эффект:
- Если remaining < 45 дней → expiry обновляется до `now() + 90 дней`.
- Если remaining ≥ 45 дней → no-op (избегаем write-нагрузки на каждый sync).
- Revoked / expired sessions **никогда не resurrect'ятся** (защита через `WHERE revoked_at IS NULL AND expires_at > datetime('now')`).

**Следствия для offline-сценариев:**

| Offline-период до возврата | Что происходит |
|---|---|
| ≤ 45 дней | Sliding refresh не сработает (remaining ≥ 50%), но session всё ещё валидна → /sync proceeds normally. |
| 45-90 дней | Первый /sync после возврата: verify passes (remaining > 0) → handler вызывает sliding refresh → expiry бампается до `now()+90d`. Эффективно cycle restart. |
| ≥ 90 дней | Session expired. `session::verify` отвергает 401. Client должен re-enroll'ить через admin-issued QR-payload. |

Если устройство **online периодически** (хотя бы раз в 45 дней), session живёт неограниченно долго — sliding refresh продлевает её на каждом sync'е.

### Push-команды (накопленные admin pushes)

`push_messages` строки сохраняются **бесконечно** пока admin не отменит `cancelled` или client не ack'нет `delivered`/`failed`. Нет auto-GC по возрасту.

**Следствие:** admin может отправить десяток `update-config` / `fetch-encrypted-file` команд в течение месяца, пока устройство offline. Когда устройство возвращается, оно их получает все на ближайшем `/sync` (LIMIT 50 за вызов — большие очереди дренируются за несколько back-to-back syncs).

### Encrypted distribution blobs

`encrypted_distributions` rows + on-disk ciphertext в `app_files_dir/encrypted/{sha256}.bin` живут до:
- `expires_at + gc_grace_days` (default 7 дней) — если admin задал `expires_at`.
- **Forever** — если `expires_at = NULL`.

`distribute_gc` background task (24h tick) удаляет purged rows + blob если на тот же sha256 не осталось active row'ов.

**Рекомендация для long-offline сценариев:** при `POST /api/v1/files/{id}/distribute` либо не передавать `expires_at`, либо ставить его в far-future (например через год от даты публикации) — иначе устройство, отсутствовавшее месяц, может вернуться и обнаружить что blob уже purgе'нут.

### Settings sync (current_state)

`devices.current_state_json` + `current_state_version` обновляются при каждом /sync если client прислал свежее `state_version`. Никакого TTL — snapshot держится последний известный, перезаписывается при следующем online.

**Следствие:** admin Web UI на `/devices/{id}/state` для month-offline девайса показывает state на момент последнего online, плюс timestamp `current_state_seen_at` чтобы оператор понимал — данные стары.

### APK update_available

Server resolve'нет update-target на каждом /sync (pinned → fleet → canary). Если client отстал на несколько minor-версий — он получит directive прыгнуть сразу на текущий target. Не нужно «лестницы» обновлений.

## Что НЕ выдерживает offline

### Полное прогорание session (> 90 дней без единого /sync)

Если устройство в hibernate'е больше 90 дней — `session::verify` отдаст 401 на возврат online'а. Client должен re-enroll'ить через admin-issued QR-payload. Это **дизайн-ограничение**: после такого срока разумно подтвердить identity заново.

Workaround: бампать `DEVICE_TOKEN_TTL_SECS` через config (если будет нужно — добавим env-override). Или Device-Owner kiosk-mode с auto-re-enroll через MDM-token (roadmap).

### `enrollment_secret` — одноразовый

Generated by admin, обменян на device_token при первом enroll'е, **wipe'нут** из БД. Если устройство offline в момент enroll'а вернётся через год — secret уже не существует, admin перегенерирует новый. Это by design — single-use credential.

### Cloud.ru presigned URLs — TTL 7 дней (max)

QR на странице `/devices/{id}/enroll` со SigV4 presigned URL'ом валиден 7 дней (SigV4 maximum). Если страница сгенерирована admin'ом неделю назад и оператор только что приехал в поле — QR может не работать. Workaround: открыть страницу `/devices/{id}/enroll` непосредственно перед моментом сканирования.

Если кардинально решим этот gap — это пункт **C** ниже (anonymous bucket policy → QR ссылается на anonymous URL без подписи).

## C — Cloud.ru anonymous bucket policy для `apks/` prefix

**Цель:** убрать requirement подписи для `apks/*` объектов. Тогда QR на странице enrollment'а будет содержать прямую ссылку `https://s3.cloud.ru/outpost/apks/outpost-latest-debug.apk` без SigV4, и она будет валидна вечно (пока bucket policy не отозвана) — что закрывает offline-gap «QR на неделю».

**Текущее ограничение Cloud.ru:** anonymous GET на `s3.cloud.ru/outpost/apks/...` возвращает 400 «missing tenant id». Это специфика Cloud.ru — даже public-read объект требует tenant context.

**Рекомендуемая policy** (применять через Cloud.ru console под write-level кредами; read-only IAM `outpost-app-readonly` не имеет permission'ов на изменение bucket policy):

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "PublicReadApksOnly",
      "Effect": "Allow",
      "Principal": "*",
      "Action": [
        "s3:GetObject",
        "s3:HeadObject"
      ],
      "Resource": [
        "arn:s3:::outpost/apks/*"
      ]
    }
  ]
}
```

**Применение:**

1. Зайти в [Cloud.ru console](https://console.cloud.ru) → Object Storage → bucket `outpost` → Permissions → Bucket policy.
2. Вставить JSON выше. Применить.
3. Verify: с любой машины без cred'ов: `curl -I https://s3.cloud.ru/outpost/apks/outpost-latest-debug.apk` → должен вернуть `200 OK` (или `403/404` если файл отсутствует, но **не** `400 missing tenant id`).
4. Если 400 продолжает приходить — Cloud.ru может требовать дополнительный заголовок типа `X-Amz-Tenant: <id>` для anonymous read. Это документировано не везде. Тогда подход не работает, остаёмся на presigned URL.

**Что менять на сервере после успешной policy:**

В `routes/web.rs::render_device_enroll` (примерно строка 1955) сейчас:

```rust
let url = signer.presigned_get_url_at(&state.cloudru_apk_key, TTL_SECS, now);
```

Заменить на:

```rust
let url = format!("https://s3.cloud.ru/{}/{}", state.cloudru_bucket, state.cloudru_apk_key);
```

Плюс убрать поля `apk_url_expires_human` из template'а (URL вечный). Это +20 строк правок, обратно совместимо (если policy откатим — переключимся обратно на presigner).

**Решение когда делать:** после того как ты применишь policy и убедишься что anonymous-GET работает. До этого presigned URL остаётся defaults — оператор обновляет страницу когда нужно дать свежий QR (типичный workflow и так укладывается в 7-дневный TTL).

## Что нужно от AR Hud команды (client-side)

Это не моя зона, но фиксирую для координации.

1. **Local-first для month-offline UX:** убедиться что local LLM (`qwen3-soldier-v24`), STT (`whisper-large-v3-turbo`), embeddings, knowledge.db — всё это on-device и работает без сети. Если хотя бы один компонент требует сеть для inference — month-offline scenario сломан.

2. **Telemetry queue retention:** OTLP-batch'и при отсутствии сети должны накапливаться в local SQLite / file queue с разумным размером. На сервер при возврате online имеют смысл события за последние ~30 дней; старее можно сбрасывать (либо отдавать пачкой через `OTel-Backfill` flag).

3. **`/api/v1/sync` long-poll support (v0.17):** при следующем удобном случае поддержать query-param `wait_for_command_ms=30000` — это даст sub-30-секундную latency для admin push'ей пока устройство в сети. Опционально (текущий immediate-return флоу работает без изменений), но улучшает UX.

4. **Token expiry detection:** при получении HTTP 401 от /sync → не паниковать, а показать в UI «Подключение к MDM истекло, нужен новый QR от админа» с глубокой ссылкой в Settings → Enroll. Сейчас по логам похоже что обработка есть, но verify нелишне.
